## Summary
A small, clean, single-purpose Arabic shipping-document extraction CLI. The code is readable and correctly structured (ABC + two extractors), secrets are properly gitignored and not tracked, and git history is clean. The real problems are operational rather than security: a batch loop that makes paid LLM calls with no rate limiting / retry / resumability, fully unpinned dependencies that threaten reproducibility, documentation drift in main.py advertising flags it does not implement, and a duplicated EXCEL_COLUMNS map that is never used. No critical security exposure was found (the live .env is gitignored and untracked). Overall grade: B

## High
### [H1] Batch loop makes paid per-PDF LLM calls with no rate limiting, retry/backoff, or resumability
- Where: ShippingDocExtractor/batch_processor.py:114-152 (process_batch loop)
- Evidence: The loop iterates `pdfs` calling `extractor.extract(pdf_path)` sequentially; there is no `time.sleep`, no retry/backoff, and results are only written to JSON/Excel once after the entire loop completes (lines 134-144).
- Impact: A transient 429/5xx or auth error mid-run silently becomes a per-file `{"error": ...}` dict (captured at line 121-132) - it is not retried, so paid extractions are lost. If the process is interrupted before the final write, every successful (paid) result in the run is discarded because nothing is persisted incrementally and there is no resume/skip-already-done logic. Each PDF is a billable Gemini/Claude vision call, so failures and interruptions cost money with no recovery.
- Fix: Persist results incrementally (append to JSON/CSV per file, or skip files already present in the output). Add a small politeness delay between calls and a bounded retry with exponential backoff for retryable errors (rate-limit / transient server errors) so paid work is not silently dropped.

## Medium
### [M1] Dependencies are completely unpinned (only `>=` lower bounds)
- Where: ShippingDocExtractor/requirements.txt:4-8
- Evidence: `google-generativeai>=0.8.0`, `anthropic>=0.40.0`, `python-dotenv>=1.0.0`, `pandas>=2.0.0`, `openpyxl>=3.1.0`. No upper bounds, no lockfile, no pyproject.
- Impact: A fresh `pip install -r requirements.txt` can pull breaking major versions. The `google-generativeai` SDK in particular has shifted APIs (it is being superseded by `google-genai`), and `anthropic` evolves quickly - either could break `genai.GenerativeModel` / `client.messages.create` usage. The build is not reproducible.
- Fix: Pin to known-good versions (e.g. `==` or compatible-release `~=`) or commit a `pip freeze` lockfile / requirements.lock generated from the working venv.

### [M2] main.py docstring/usage advertises `--batch --excel` flags it does not implement
- Where: ShippingDocExtractor/main.py:1-8 (module docstring, line 6)
- Evidence: Docstring shows `python main.py --input ./folder/ --batch --excel output.xlsx`, but `main()` (lines 50-83) only defines `--input/--output/--api/--print`. Batch and Excel live in batch_processor.py.
- Impact: A user following the documented command gets `unrecognized arguments: --batch --excel` and may think the tool is broken. Documentation drift erodes trust in the rest of the docs.
- Fix: Remove the `--batch --excel` line from main.py's docstring and point users to `batch_processor.py` for batch/Excel workflows.

### [M3] Duplicated `get_extractor()` across two entry points
- Where: ShippingDocExtractor/main.py:19-26 and ShippingDocExtractor/batch_processor.py:26-33
- Evidence: The two functions are byte-for-byte identical (gemini/claude/else->ValueError).
- Impact: Maintenance hazard - adding a new API/provider or changing dispatch requires editing both copies; they can silently drift.
- Fix: Move `get_extractor()` into a shared module (e.g. extractors/__init__.py as a factory) and import it from both entry points.

### [M4] `EXCEL_COLUMNS` map in prompts.py is dead code; flatten_result hardcodes Arabic headers
- Where: ShippingDocExtractor/prompts.py:77-87 (EXCEL_COLUMNS) vs ShippingDocExtractor/batch_processor.py:17 (import) and 51-79 (flatten_result hardcoded dict literals)
- Evidence: `from prompts import EXCEL_COLUMNS` is imported but never referenced; `flatten_result` writes Arabic header strings inline (e.g. `"التاريخ"`, `"رقم الحاوية"`). The two definitions can diverge - and already partially do (EXCEL_COLUMNS has no "العملة"/currency and no "ملف المصدر"/source-file column, while flatten_result emits both).
- Impact: A maintainer editing EXCEL_COLUMNS expecting it to change the Excel output will see no effect; the unused import is misleading and the duplicated header strings can drift.
- Fix: Either drive `flatten_result` from `EXCEL_COLUMNS` (single source of truth) or delete the unused map and import.

### [M5] Broad `except Exception` swallows API errors into result dicts, masking auth/rate-limit failures
- Where: ShippingDocExtractor/extractors/gemini_extractor.py:55-60, ShippingDocExtractor/extractors/claude_extractor.py:75-80, ShippingDocExtractor/batch_processor.py:127-132
- Evidence: Each extractor wraps the API call in `except Exception as e: return {"error": str(e), ...}`, and the batch loop truncates the message to 50 chars (`{result.get('error','...')[:50]}`).
- Impact: A 401 (bad key), 429 (rate limit), or quota-exhausted error is indistinguishable from a per-document parse failure. In batch mode every file can fail identically (e.g. invalid key) while the run reports "0/N successful" with truncated messages and exits 0, wasting time and obscuring the real cause. There is also no JSON-schema validation of the model output beyond `json.loads`.
- Fix: Distinguish fatal/auth/rate-limit exceptions (abort the batch or surface full message + exit non-zero) from per-document errors; avoid truncating the diagnostic message, or log full errors to a side file.

## Low
### [L1] No tests exist
- Where: ShippingDocExtractor/ (no tests/ dir, no test files)
- Evidence: `git ls-files` lists only source/config; CLAUDE.md states "No tests exist."
- Impact: Pure, side-effect-free logic like `BaseExtractor._parse_json_response()` (markdown-fence stripping) and `flatten_result()` (containers vs no-containers rows) is easy to regress unnoticed.
- Fix: Add a minimal pytest covering `_parse_json_response` (```json fenced, bare ```, plain JSON, malformed) and `flatten_result` (empty vs multi-container).

### [L2] Generated business data committed/present at repo root (gitignored but on disk)

### [L3] README install step uses Windows-only `copy` and docs assume a specific Scans file count
- Where: ShippingDocExtractor/README.md:24 (`copy .env.example .env`) and README.md:92-98 ("Process the 8 PDF files")
- Evidence: `copy` is a Windows/cmd command; on the Linux environment here it is `cp`. The "8 PDF files" count is hardcoded.
- Impact: Cross-platform friction for a Linux user (the documented copy command fails) and a brittle, stale count.
- Fix: Use `cp .env.example .env` (or show both), and describe the input generically ("the PDFs in the Scans folder").

## Optimizations
- Persist batch progress incrementally and add a `--resume`/skip-existing option keyed on `_source_file`, so an interrupted paid run does not lose completed extractions (overlaps with H1).
- Consolidate the provider-dispatch factory (`get_extractor`) and consider a single shared `cli_common` for argparse boilerplate shared by the two entry points.
- Add lightweight schema validation of the model's JSON (expected keys/types) before writing Excel rows, so malformed extractions are flagged rather than silently producing rows of `None`.
- Consider migrating `google-generativeai` to the newer `google-genai` SDK before the legacy package is deprecated; pin versions either way.
